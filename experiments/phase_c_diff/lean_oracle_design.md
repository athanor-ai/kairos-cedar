# Cedar Spec Lean evaluator as third oracle

## Entry point used

`Cedar.Spec.isAuthorized` from `cedar-spec/cedar-lean/Cedar/Spec/Authorizer.lean`:

```lean
def isAuthorized (req : Request) (entities : Entities) (policies : Policies) : Response
```

`Response.decision : Decision` where `Decision` is `allow | deny` and derives
`Lean.ToJson`, so the output can be printed directly.

The oracle does **not** round-trip through Cedar text format. It evaluates the
Lean AST values produced by `CedarFull.PolicyGen.genTuple` directly, which
means it calls the same evaluator that the cedar-spec proofs are stated over.

## How Schema and Entities are constructed

`Schema` is unused by `isAuthorized` (the authorizer takes entities, not a
schema; validation is a separate pass). The `Entities` value is constructed
in `cedar-full/CedarFull/LeanOracle.lean` as a hand-written `Map EntityUID
EntityData` that mirrors `FIXED_ENTITIES` in `run_diff.py`:

- `User::"alice|bob|carol"` each carry `address: {city, street, zip}` as a
  `Value.record`.
- `Document::"doc1|doc2"` each carry `owner: User::"alice"` as a
  `Value.prim (.entityUID ...)`.
- `Photo::"photo1"` and the three `Action::*` entities carry empty attribute
  maps.
- All ancestor sets and tag maps are empty (the generator policies do not
  exercise hierarchy or tags).

## Lean binary: location and invocation

Lake exe target `measure-lean` is declared in `cedar-full/lakefile.toml`:

```toml
[[lean_exe]]
name = "measure-lean"
root = "MeasureLean"
```

After `lake build` inside the container the binary lives at:

```
cedar-full/.lake/build/bin/measure-lean
```

It accepts one argument `n` (number of tuples) and writes one JSON line per tuple
to stdout:

```
{"idx": "0", "decision": "Allow"}
{"idx": "1", "decision": "Deny"}
```

If the binary is not yet built, `lean_runner.py` falls back to:

```bash
cd cedar-full && lake env lean --run MeasureLean.lean <n>
```

## Python wrapper

`experiments/phase_c_diff/lean_runner.py` exposes:

```python
from experiments.phase_c_diff.lean_runner import run_lean_batch, build_lean_oracle

# Build once (optional; run_lean_batch auto-falls-back if binary absent)
build_lean_oracle(timeout=600)

# Evaluate n tuples; returns dict[idx_str -> "Allow" | "Deny" | "ERROR(...)"]
lean_decisions = run_lean_batch(n=1000, timeout=600)
```

All container calls use the same `run_in_image` pattern as `run_diff.py`.

## Wiring --three-way into run_diff.py

When `--three-way` is passed, `run_diff.py` imports `lean_runner` and adds a
fourth phase after the Go batch:

```
[4c] Lean oracle (measure-lean) ...
```

The summary section adds:
- lean vs rust agreement rate
- lean vs go agreement rate
- three-way agreement rate (all three match)
- three-way disagreements (tuples where at least one oracle diverges)

Index alignment: `measure-lean` and `measure-diff` both use the same
`genTuple` support and cycle with `i % supportSize`, so `idx` values are
directly comparable. The Python runner joins on `idx` string with no
additional mapping needed.

## Subtleties

**No JSON parsing in Lean.** Cedar's text parser lives in Rust
(`cedar-policy`) and Go (`cedar-go`); there is no Lean-native Cedar text
parser. `MeasureLean.lean` avoids this by working directly on the AST
generated by `genTuple` rather than re-parsing the policy text emitted by
`MeasureDiff.lean`. This is intentional: the Lean oracle evaluates _exactly_
the same AST the generator constructed, which is what we want for a
spec-faithful third oracle.

**Entities vs Schema.** `isAuthorized` takes `Entities` (a runtime entity
store), not `Schema` (a compile-time type annotation). The schema is used only
by the Cedar validator (`Cedar.Validation.typeOf`). Since the generator
produces well-typed policies under `fixedSchema`, we do not need to call the
validator in the oracle path.

**Decision JSON encoding.** `Decision` derives `Lean.ToJson` in
`cedar-spec/cedar-lean/Cedar/Spec/Response.lean`, yielding `"allow"` /
`"deny"` (lowercase). `lean_runner.py` normalises these to `"Allow"` /
`"Deny"` to match the casing from the Rust and Go runners.

**Build time.** `lake build measure-lean` re-elaborates `CedarFull` +
`CedarBridge` + `Cedar` from cache on first run (~2-4 min in the container);
subsequent incremental builds are fast. For CI, the `--three-way` flag should
be preceded by `build_lean_oracle()` or an explicit `lake build` step.
